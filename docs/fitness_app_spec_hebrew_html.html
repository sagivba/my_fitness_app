אפיון אפליקציית ספורט אישית - My Fitness App

1. מטרת המערכת

המערכת מיועדת לאיסוף, תיעוד וניתוח נתוני כושר, פעילות, תזונה ושינה של משתמש יחיד, עם דגש על בניית בסיס נתונים נקי, עקבי וניתן להרחבה.

בשלב הראשון המערכת לא תנסה להסיק מסקנות מורכבות או לבצע ניתוחי קורלציה מלאים. מטרת ה-MVP היא ליצור תשתית אמינה לאיסוף נתונים: אימונים, נתוני Garmin, תיעוד אוכל, שינה, תחושות, הערות ומדדים ידניים בסיסיים.

החלטת MVP: קודם בונים מערכת איסוף נתונים אישית, מקומית, פשוטה ומדויקת. ניתוח קשרים בין אימונים, אוכל ושינה ייבנה רק לאחר שיצטברו נתונים איכותיים.

2. משמעות "אפליקציה אישית שרצה מקומית"

אפליקציה מקומית היא אפליקציית Web רגילה מבחינת חוויית המשתמש, אבל היא אינה מפורסמת לאינטרנט ואינה מותקנת בשרת ציבורי. היא רצה על המחשב שלך, למשל דרך Flask או Docker, ונפתחת בדפדפן בכתובת כמו: 

http://127.0.0.1:5000

המשמעות המעשית:

  • הנתונים נשמרים אצלך, למשל בקובץ SQLite מקומי או בבסיס נתונים מקומי.
  • אין צורך בשלב הראשון במנגנון משתמשים, הרשאות, סיסמאות, SSL או deployment.
  • אפשר לפתח מהר יותר, לבדוק מהר יותר, ולשמור על פרטיות טובה יותר.
  • בהמשך ניתן להפוך אותה לאפליקציה רשתית מלאה, אם תרצה גישה מטלפון או ממחשבים אחרים.

בשלב הראשון, גם אם הממשק נראה כמו אתר, הוא מיועד לשימוש אישי בלבד.

3. קהל יעד והנחות עבודה

  • משתמש יחיד.
  • שעון Garmin Vivoactive.
  • אימונים עיקריים: אימוני כוח בחוץ, אימון מכון משולב אירובי וכוח, אליפטיקל, הליכה, אופניים.
  • משך אימון טיפוסי: 30 עד 60 דקות.
  • בשלב ראשון: איסוף נתונים; בשלב שני: ניתוח מגמות וקשרים.
  • טכנולוגיה מועדפת: Flask פשוט, HTML, CSS, מעט JavaScript לפי צורך.
  • בדיקות: Python unittest בלבד.

4. בחירת טכנולוגיה

4.1 החלטה: Flask

הבחירה המומלצת לשלב הראשון היא Flask.

הסיבות:

  • השלד הקיים כבר מתאים ל-Flask.
  • המערכת היא קודם כל מערכת איסוף נתונים אישית, לא API ציבורי מורכב.
  • אפשר לבנות מסכי HTML server-side פשוטים ומהירים.
  • קל לשמור על ארכיטקטורה פשוטה: routes -> services -> model.
  • קל לבדוק עם unittest בלי להכניס stack כבד.
  • קל להעביר משימות קטנות ל-Codex.

4.2 מתי FastAPI כן יהיה עדיף בעתיד

FastAPI יהיה מתאים יותר אם בעתיד המערכת תהפוך ל-API-first, למשל:

  • אפליקציית Mobile נפרדת.
  • אינטגרציות רבות מול שירותים חיצוניים.
  • שימוש כבד ב-async.
  • צרכני API מרובים.
  • OpenAPI schema כמוצר מרכזי.

לשלב הנוכחי אין סיבה טובה להעדיף FastAPI על Flask.

5. התאמת השלד הקיים

הריפו הקיים מבוסס על תבנית Python ידידותית ל-AI. משימת Codex הראשונה צריכה להיות התאמת השלד לפרויקט, לפני פיתוח פיצ'רים.

5.1 שינויים נדרשים בשלד

  • להחליף שם פרויקט מ-template לשם האפליקציה.
  • להחליף package פנימי ל-my_fitness_app.
  • לעדכן README, pyproject, Docker Compose, AGENTS, docs ופקודות הרצה.
  • לשמור על מבנה routes/services/model.
  • להשאיר unittest כמדיניות בדיקות.
  • לא להוסיף תלויות כבדות ללא הצדקה.
החלטת שם: שם הריפו הרשמי הוא sagivba/my_fitness_app. שם ה-package הפנימי המומלץ הוא my_fitness_app, כדי שתהיה התאמה מלאה בין שם הריפו, שם הפרויקט ושם החבילה.

7. מקורות נתונים

7.1 Garmin

בשלב הראשון המערכת תתמוך בייבוא ידני מקבצים, ולא באינטגרציה רשמית ישירה. הסיבה: ייבוא קבצים מהיר יותר לפיתוח, פשוט יותר לבדיקה, ואינו דורש אישורי API.

פורמטים רצויים:

  • FIT: הפורמט העשיר והמועדף לפעילות בודדת, כולל נתונים מפורטים.
  • TCX: פורמט XML שימושי לנתוני activity, דופק, מסלול ומשך.
  • GPX: טוב בעיקר למסלול ומיקום, פחות עשיר למדדי כושר.
  • CSV: טוב לייבוא טבלאי וסיכומי split, פחות מתאים לנתונים מפורטים מלאים.

המלצה מעשית: להתחיל מייבוא CSV ידני או טופס ידני לצורך MVP, ואז להוסיף FIT כייבוא העיקרי לנתוני Garmin עשירים.

7.2 תזונה

  • קישור למתכון באתר המתכונים האישי, אם רלוונטי.
  • הערכת קלוריות, חלבון, פחמימות, שומן וסיבים באמצעות OpenAI API.
  • שמירת רמת ודאות והערות משתמש.

8. סוגי אימונים

סוג אימון מדדים עיקריים הערות
אימון כוח בחוץ משך, דופק ממוצע, דופק מקסימלי, RPE, תרגילים, סטים, חזרות, הערות. ייתכן שאין משקל חיצוני; חשוב לתעד תחושה ועומס.
מכון כושר משולב חלוקה לאירובי וכוח, משך כל חלק, מכשירים, דופק, עומס, תרגילים. אימון טיפוסי 30-60 דקות.
אליפטיקל משך, דופק, קלוריות, רמת התנגדות, cadence אם קיים, RPE. מדד טוב לאירובי ללא עומס ריצה.
הליכה משך, מרחק, קצב, צעדים, דופק, עליות, מסלול. חשוב להבחין בין הליכה רגילה להליכה כאימון.
אופניים משך, מרחק, מהירות, דופק, עליות, cadence אם קיים. רלוונטי גם לאופניים נייחים וגם בחוץ.

13. ניתוח קשרים - שלב עתידי

לאחר שיצטברו נתונים, ניתן להוסיף שכבת ניתוח.

  • קשר בין סוג אימון לבין איכות שינה באותו לילה.
  • קשר בין שעת אימון לבין שינה.
  • קשר בין ארוחה מאוחרת לבין איכות שינה.
  • קשר בין עומס אימונים שבועי לבין עייפות.
  • קשר בין משך שינה לבין ביצועי אימון ביום שאחרי.
  • קשר בין תזונה לבין תחושת אנרגיה.
